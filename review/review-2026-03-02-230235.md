## Summary

The codebase is well-structured with solid error-handling discipline, a clean Pydantic model layer, and good defensive coding at the workflow boundary. The contract review pipeline is the most complex area and contains several logic bugs that affect result correctness: a dead `result` variable that silently produces empty clause evidence lists, path-resolution that uses relative rather than absolute paths (breaking any worker-process deployment), and a regex-injection vector in the frontend keyword highlighter. Medium-severity issues include keyword-set overlaps that inflate detection scores, uncovered edge cases in page unpacking and sorting, and a confusing boolean logic inversion in `_is_likely_operative_contract`. Overall quality is high for an MVP but several of the bugs are production-blocking.

**C-01 — Dead `result` variable makes all "missing clause" risk items carry empty evidence**
File: `backend/services/contract_review_service.py:1904`

Problem: Line 1904 builds `result` from `presence_map[clause_type]` but it is never used. Two lines later `clause_ids` is unconditionally set to `[]` and `page_numbers` to `[]`. The intent was clearly to populate `clause_ids` and `page_numbers` from `result`, but the code never reads `result` again. Every "not-detected" risk item therefore has empty `clause_ids` and `page_numbers`, making PDF evidence links and page citations dead for all missing-clause risks.

```python
# line 1904 — result is constructed but never consumed
result = {"clause_id": (entry.get("clause_ids") or [None])[0], "page_number": (entry.get("page_numbers") or [None])[0]}
severity = risk_weights.get(clause_type, "medium")
...
clause_ids = []          # ← always empty; result.clause_id discarded
page_numbers = []        # ← always empty; result.page_number discarded
```

Fix:
```python
clause_ids = [result["clause_id"]] if result.get("clause_id") else []
page_numbers = [result["page_number"]] if result.get("page_number") else []
```

**C-02 — Relative YAML paths break when CWD differs from project root**
File: `backend/services/contract_review_service.py:1077,1088`

Problem: Both YAML loaders use plain relative paths (`Path("backend/contract_profiles/risk_explanations.yaml")` and `Path("backend/legal_references/jurisdiction_statutes.yaml")`). These resolve against the process CWD, so the service silently returns `{}` in any deployment where the server is not launched from the exact project root. Streamlit workers and some ASGI configurations start from a different directory. The companion YAML loader in `clause_taxonomy.py` correctly uses `Path(__file__).parent.parent / ...`.

Fix: Replace both loaders with absolute file-relative paths:
```python
Path(__file__).parent.parent / "contract_profiles" / "risk_explanations.yaml"
Path(__file__).parent.parent / "legal_references" / "jurisdiction_statutes.yaml"
```

**C-03 — ReDoS / regex-injection via `matched_keyword` in the frontend highlighter**
File: `frontend/app.py:763-768`

Problem: The matched keyword from the backend is passed into `re.escape(kw)` — which is correct — but `kw` itself can contain regex metacharacters when it comes from the `_find_hits` pattern-matching code that stores `.pattern` strings (e.g. `r"\bwithout notice\b"`). When `re.escape` is applied they produce double-escaped output; if the backend ever changes to store pattern strings directly in `matched_keyword`, the `_re.sub` call would execute them as regex against user-visible text — a code-execution gadget path.

```python
# app.py ~762-768
text = _re.sub(
    r"(?i)(" + _re.escape(kw) + r")",   # ← kw unvalidated; could be a regex pattern
    r"**\1**",
    text,
    count=1,
)
```

Fix: Validate that `kw` is plain text (no regex metacharacters), or switch to `str.replace` for keyword emphasis so no regex engine is involved.

**H-01 — `_page_fallback_scan` unpacks pages as 2-tuples but OCR pages are 3-tuples**
File: `backend/services/contract_review_service.py:1000`

Problem: `for page_text, page_num in pages:` raises `ValueError: too many values to unpack` on OCR pages, which are 3-tuples `(text, page_num, is_ocr)`. The exception is swallowed by the outer `try/except`, making the entire fallback scan return `{}` silently. Clauses in scanned pages will always remain `not_detected`.

Fix: Mirror the guard from `structured_clause_extraction.py:302-306`:
```python
if len(p) == 3:
    text, page_num, _ = p
else:
    text, page_num = p
```

**H-02 — `_is_likely_operative_contract` intent mismatch: guard rarely fires**
File: `backend/services/contract_review_service.py:438`

Problem: The docstring says "No parties / no agreement in lead" but the condition uses `and` rather than `or`, so it only fires when BOTH parties language AND "agreement" are absent simultaneously. Documents without the literal word "agreement" (common in Arabic/bilingual translations) that do mention "employer" will not trigger the warning. The intended guard (fire if _either_ signal is missing) requires `or`.

Fix:
```python
# intended: fire if no agreement word AND no party terms
if "agreement" not in lead or ("employer" not in lead and "employee" not in lead):
    return False
```

**H-03 — `_find_document_path` glob fallback can silently return the wrong file**
File: `backend/services/contract_review_service.py:396-405`

Problem: `glob(f"{document_id}*{ext}")` matches any file whose name starts with `document_id`. If the store contains both `DOC-001.pdf` and `DOC-001-amended.pdf`, the fallback returns `matches[0]` with no ordering guarantee. This affects both the page-fallback scan and the main `run()` document-classification call.

Fix: Require exact-match only; fail explicitly rather than falling back to glob.

**H-04 — `classify_clause_type_string` double-counts scores for slugs present in both enum-backed and YAML keyword sets**
File: `backend/services/clause_taxonomy.py:226-229`

Problem: When a slug like `"governing_law"` appears in both `category_keywords` (via `ClauseType` enum iteration) and `string_type_keywords` (from the YAML), both loops add to `all_scores[slug]`, doubling the score and biasing results toward clause types that have both an enum and a YAML entry.

Fix: Exclude from `string_type_keywords` any slug already covered by an enum-backed entry, or use a single unified scoring pass.

**M-02 — `_is_administrative_page` pattern `\bno\.\s*\d+\b` matches common legal text**
File: `backend/services/structured_clause_extraction.py:248`

`"Article No. 5"`, `"Clause No. 3"`, `"Reference No. 12"` all match this pattern and cause entire pages to be classified as `ADMINISTRATIVE_MATERIAL`, silently excluding them from clause extraction.

Fix: Scope the pattern to known administrative prefixes (e.g. `"Ministry No."`, `"Circular No."`) rather than bare `"no. N"`.

**M-03 — `detect_document_type` classifies any document mentioning "party" as a CONTRACT**
File: `backend/services/structured_clause_extraction.py:923`

`"party"` is extremely common in judgments and statutes. A single occurrence causes `CONTRACT` classification, enabling the contract entry gate and clause extraction on documents that should not be processed as contracts.

Fix: Require two signals for CONTRACT detection (e.g. both `"party"` and `"agreement"`, or `"employer"` and `"employee"`).

**M-04 — `_aggregate_status` priority places `"uncertain"` above `"implicitly_covered"`, producing misleading executive-summary text for mixed groups**

When a canonical group has one `"implicitly_covered"` clause and one `"uncertain"` clause, the aggregate returns `"uncertain"`, and the observation text reads "commonly expected details not clearly specified" — not the same as "implicitly covered." This is misleading legal output.

**M-05 — `"yes" in answer` Ollama binary check matches "yes" anywhere in a longer response**

`"Not necessarily yes, ..."` would trigger a `"detected"` result. Use `answer.startswith("yes")` or `answer == "yes"` for strict binary detection.

## Positive Notes

- The Pydantic model layer (`contract_review.py`) is clean and uses `default_factory` correctly throughout — no mutable default argument bugs.
- Error handling at the workflow boundary is disciplined: `ctx.fail()` is called consistently with structured error codes.
- All YAML deserialization uses `yaml.safe_load`, never the unsafe `yaml.load`.
- `clause_taxonomy.py` YAML loader correctly uses `Path(__file__).parent.parent` for file-relative resolution — the pattern `contract_review_service.py` should copy.
- The two-pass LLM guardrail (`_resolve_uncertain_with_llm`) is fully exception-isolated and never bubbles failures to the caller.
- `_normalize_text` is reused consistently for all keyword matching, preventing case/punctuation drift across the pipeline.
- The frontend's delete-confirmation (two-click) pattern is a good UX safety guard.
- `_deduplicate_clauses` uses a `(section, verbatim_text)` composite key, correctly allowing the same text in different sections.
