**Date:** 2026-03-03
**Reviewer:** Claude Sonnet 4.6 (automated)
**Scope:** Arabic PDF download implementation
**Files reviewed:**
- `backend/main.py` — `POST /api/translate` endpoint (lines 1324–1342)

---

## Summary

The implementation correctly achieves the goal: a second "Download in Arabic" button appears next to the English PDF button, Arabic text is reshaped via `arabic_reshaper`+`python-bidi`, all dynamic strings are translated in one batch call, and the feature degrades gracefully if translation or font registration fails.

**However, there are 3 functional bugs** (HTML markup corruption, untranslated severity/status values, and duplicate font registration crash) that will affect rendering quality or cause errors in real use. There are also minor security and maintainability concerns.

---

## Findings

### 🔴 Bug 1 — `_ar()` corrupts HTML markup in Paragraph strings
**File:** `frontend/app.py`
**Lines:** 1354, 1469, 1471

**Problem:** Several calls pass HTML-tagged strings into `_ar()`:
```python
_ar(f"<b>{_AR[section_title_key]}</b>")   # line 1354
_ar(f"<b>{r.get('description', '')}</b>")  # line 1469
_ar(f"<i>{r['severity_reason']}</i>")      # line 1471
```
`arabic_reshaper.reshape()` and `get_display()` treat the entire string as a character sequence. The bidi algorithm will mirror the RTL content and can reorder the `<b>`, `</b>` tag characters relative to the Arabic text, producing malformed markup like `">b/<جئاتنلا>b<"`. ReportLab's Paragraph XML parser will fail to render bold/italic, silently emitting plain text or raising a parse error.

**Fix:** Apply `_ar()` to the content only, then wrap in tags:
```python
# Instead of:
_ar(f"<b>{content}</b>")
# Use:
f"<b>{_ar(content)}</b>"
```
Affected lines: 1354, 1469, 1471.

### 🔴 Bug 2 — Font re-registration crash on every Streamlit rerender
**File:** `frontend/app.py`
**Lines:** 1173–1177

```python
pdfmetrics.registerFont(TTFont("Arabic", font_path))
AR_FONT = "Arabic"
```
Streamlit reruns the entire script on every interaction. On the second render of `pdf_pending`, this line attempts to register the font `"Arabic"` again. ReportLab will raise an error on duplicate registration in some versions (`KeyError` or `AssertionError`), causing the Arabic PDF generation to fail.

**Fix:** Guard the registration:
```python
if "Arabic" not in pdfmetrics.getRegisteredFontNames():
    pdfmetrics.registerFont(TTFont("Arabic", font_path))
AR_FONT = "Arabic"
```

### 🟡 Bug 3 — Severity and status values appear in English in the Arabic PDF
**File:** `frontend/app.py`
**Lines:** 1375, 1418

**Clause Coverage table** (line 1375):
```python
sev_text = (item.get("severity") or "").capitalize() or "—"
# produces "High", "Medium", "Low" — English
```
**Risk Analysis table** (line 1418):
```python
Paragraph(_ar(r.get("status", "").replace("_", " ").title()), ar_cell)
# produces "Risk", "Finding", "Confirmation" — English
```
Neither field is included in `dyn_strings` for translation, and neither is mapped through `_AR`. The `_AR` dict already has `"high"/"medium"/"low"` and `"risk_status"/"finding"/"confirmed"` keys but they are not used here.

**Fix for severity** (line 1375):
```python
sev_raw = (item.get("severity") or "").lower()
sev_text = _AR.get(sev_raw, sev_raw.capitalize()) or "—"
```
**Fix for status** (line 1418):
```python
status_raw = r.get("status", "").lower().replace(" ", "_")
status_map = {"risk": _AR["risk_status"], "finding": _AR["finding"], "confirmation": _AR["confirmed"]}
Paragraph(_ar(status_map.get(status_raw, r.get("status", ""))), ar_cell)
```

### 🟡 Issue 4 — `/api/translate` has no input size guard
**File:** `backend/main.py`
**Lines:** 1324–1342

The endpoint accepts an arbitrary JSON list with no size validation:
```python
parsed = _json.loads(texts)
for t in parsed:
    results.append(translation_service.translate_text(...))
```
A large contract with many risks and snippets could send 100–200 strings. More critically, a malformed or malicious request could send thousands of strings, generating unbounded Google Translate API calls synchronously in a single request. There is also no `try/except` around `_json.loads(texts)`, so a malformed `texts` field raises an unhandled `JSONDecodeError` (FastAPI returns a 500 instead of a 422).

**Fixes:**
```python
try:
    parsed = _json.loads(texts)
except _json.JSONDecodeError:
    raise HTTPException(status_code=422, detail="texts must be a valid JSON array")
if not isinstance(parsed, list):
    raise HTTPException(status_code=422, detail="texts must be a JSON array")
if len(parsed) > 500:
    raise HTTPException(status_code=422, detail="Too many strings (max 500)")
```

### 🟡 Issue 5 — Silent translation failure gives no user feedback
**File:** `frontend/app.py`
**Lines:** 1112–1113

```python
except Exception:
    return texts  # fallback: untranslated
```
When the backend translation call fails (503, timeout, network error), `_translate_texts` silently returns the original English strings. The user sees the Arabic PDF button, clicks download, and gets an English-text PDF with Arabic headings — with no warning that translation failed.

**Fix:** Log the exception at minimum:
```python
except Exception as e:
    import logging
    logging.getLogger(__name__).warning(f"Arabic translation failed: {e}")
    return texts
```
Optionally surface a Streamlit warning via a shared state flag set before/after the call.

### 🟠 Issue 6 — Maintenance burden: `_generate_arabic_pdf_buffer` duplicates 350+ lines
**File:** `frontend/app.py`
**Lines:** 1154–1528

The Arabic PDF function is a near-complete copy of `_generate_pdf_buffer` with `_ar()` calls added and styles changed. Any structural change to the English PDF (new section, changed layout) must be manually replicated. This is acceptable for the current scope but will become a maintenance problem.

**Suggested future refactor:** Extract a shared `_build_pdf_story(resp, styles, cell_style, header_cell_style, ar_fn=None)` that both functions call, parameterised by style objects.

---

### 🟢 Issue 7 — `_AR`, `_ar`, `_translate_texts` redefined on every Streamlit rerender
**File:** `frontend/app.py`
**Lines:** 1091–1152

All helper functions and the `_AR` dict are re-created on every Streamlit script execution (every user interaction). This is a cosmetic Streamlit pattern issue — the functions are lightweight, so it has no runtime impact. However, it would be cleaner to define these at module level.

## Positive Notes

1. **Collect-once, translate-once pattern** is well-designed. All dynamic strings are gathered into a single flat list and sent in one `POST /api/translate` call, minimising network round trips and avoiding per-field latency.

2. **`_t()` closure** with `nonlocal idx` is a clean and readable approach for maintaining positional synchrony between the collection and reconstruction passes. The guard `if idx < len(translated)` prevents index errors if the translation call returns a shorter list.

3. **Symmetric traversal** between `dyn_strings` collection and the reconstruction block — they follow the same loop structure and filters (`isinstance(item, dict)` etc.) in the same order. This is maintainable and correct.

4. **Graceful degradation at every level:**
   - Font unavailable → Helvetica fallback
   - `arabic_reshaper` not installed → raw string passthrough
   - Translation backend down → English text passthrough
   - Each layer fails independently without crashing the PDF build.

5. **Row colors are preserved correctly.** Severity-based row coloring uses the original `resp` severity field (e.g. `"high"`, `"medium"`) rather than the translated label, so colour logic is unaffected by translation.

6. **The two-column UI layout** (`col_en, col_ar = st.columns(2)`) is clean and the spinner scoping (`with col_ar: with st.spinner(...)`) ensures the spinner shows only during Arabic generation.
