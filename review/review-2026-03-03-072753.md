## Summary

The classification pipeline has a well-structured two-stage architecture (Ollama LLM → DistilBERT) for document ingestion and a rich keyword taxonomy for clause coverage detection. However, there is a critical **integration gap** between the two systems: the Ollama-detected `contract_type` is stored in the database but never fed back into the contract review workflow, meaning the system always requires the user to manually specify type. Several additional logic bugs and keyword conflicts degrade classification accuracy.

## Findings

### CRITICAL

---

#### F-01 — `detected_contract_type` is never used to default the review profile
**File:** `backend/main.py:1005`, `backend/services/contract_review_service.py:1552–1557`
**Category:** Logic / Integration

`DocumentClassificationService.classify()` correctly detects `contract_type` (employment/nda/msa) during upload via Ollama and stores it in the document registry with `save_classification()`. The `/api/documents/{document_id}/classification` endpoint exposes this. However, when `/api/contract-review` is called, `contract_type` is read exclusively from the user's form input:

```python
# main.py:1005
contract_type: str = Form("employment")  # hardcoded default
```

```python
# contract_review_service.py:1552–1557
contract_type_key = meta.get("contract_type")
if not contract_type_key:
    _log.warning("No contract_type in metadata ... — defaulting to 'employment'")
    contract_type_key = "employment"
```

The stored `detected_contract_type` is **never consulted** as a fallback. An NDA uploaded (correctly classified by Ollama as "nda") will be reviewed as an employment contract if the user doesn't change the dropdown, producing entirely wrong clause gap results.

**Impact:** High — misleading reviews for users who don't know to change the contract type.

#### F-02 — `"token"`, `"chunk"`, and `"embedding"` in page text silently drop all clauses
**File:** `backend/services/structured_clause_extraction.py:315–318`
**Category:** Logic / Edge Case

```python
if text and ('embedding' in text.lower() or 'chunk' in text.lower() or 'token' in text.lower()):
    logger.error("Input appears to be chunks/embeddings, not page-level text. Fail-closed.")
    return []
```

This check rejects the entire document if any page contains these words literally. Real contracts frequently contain:
- "token" — digital token agreements, blockchain IP clauses
- "chunk" — "a chunk of liability", "chunk of the payment"
- "embedding" — embedding intellectual property rights

If such a page exists, `extract_structured_clauses()` returns `[]` silently (only an error log), and the review engine gets no clauses to match against.

**Impact:** High — complete clause extraction failure on legitimate contracts.

#### F-03 — `detect_document_type()` detects STATUTE on any page with "section"
**File:** `backend/services/structured_clause_extraction.py:919–929`
**Category:** Logic

```python
def detect_document_type(self, pages: List[str]) -> DocumentType:
    sample = " ".join(pages[:self.MAX_DOC_TYPE_SAMPLE_PAGES]).lower()
    if "judgment" in sample or "lord" in sample or "court of appeal" in sample:
        return DocumentType.JUDGMENT
    if "this agreement" in sample or ("party" in sample and "agreement" in sample):
        return DocumentType.CONTRACT
    if "an act to" in sample or "section" in sample:   # ← BUG
        return DocumentType.STATUTE
    return DocumentType.UNKNOWN
```

`"section"` alone triggers STATUTE detection. The word "section" appears in virtually every legal document ("Section 3.2", "this section", "in the section below"). This branch is only reached if the CONTRACT check fails, but Arabic-first contracts with no "this agreement" in the first 3 pages would fall through to STATUTE. Once typed as STATUTE, judgment-specific branch logic is skipped, but the document never gets typed as CONTRACT, degrading clause extraction quality.

**Impact:** Medium-High — Arab-language or unusual preamble contracts mis-typed as STATUTE.

**File:** `backend/services/contract_review_service.py:1555–1557`
**Category:** Logic / Assumptions

#### F-05 — `"other"` contract type has no profile → `PROFILE_NOT_FOUND` crash
**File:** `backend/services/document_classification_service.py:356`
**Category:** Edge Case

The Ollama classification prompt explicitly lists `"other"` as a valid `contract_type`:
```
"contract_type": "employment" | "nda" | "msa" | "other" | null
```

But there is no `backend/contract_profiles/other.yaml`. If the LLM returns `contract_type="other"` and the user submits the review form without changing it, `load_contract_profile("other")` raises `FileNotFoundError`, and the workflow fails with `PROFILE_NOT_FOUND`.

**Impact:** Hard crash for any document the LLM thinks is a "generic" contract.

#### F-07 — Email regex in `administrative_markers` silences Notices sections
**File:** `backend/services/structured_clause_extraction.py:247`
**Category:** Logic / Edge Case

```python
self.administrative_markers = [
    ...
    r"@\w+\.\w+",   # email regex
    ...
]
```

Any page containing an email address is classified as `ADMINISTRATIVE_MATERIAL` and excluded from clause extraction. "Notices" sections in contracts routinely include email addresses for service (e.g., `notice@company.com`). This causes the entire Notices section — which often contains governing_law, jurisdiction, and dispute_resolution information — to be silently excluded.

**Impact:** Medium-High — governing law / jurisdiction clauses can be missed if they appear on a Notices page.

#### F-08 — `classify_clause_type()` returns `ClauseType.OTHER` for all string-type slug matches
**File:** `backend/services/clause_taxonomy.py:198–208`
**Category:** Logic / Maintainability

```python
# Fall back to string-keyed keywords (covers ip_ownership, sla_obligations, etc.)
string_scores: Dict[str, int] = {}
for slug, keywords in self.string_type_keywords.items():
    score = sum(1 for kw in keywords if kw in text_lower)
    if score > 0:
        string_scores[slug] = score

# If a string-type match found, return OTHER (caller can read normalized_clause_type)
if string_scores:
    return ClauseType.OTHER  # ← winning slug is discarded
```

When a string-type clause (ip_ownership, sla_obligations, etc.) scores highest, `classify_clause_type()` throws away the winning slug and returns `ClauseType.OTHER`. The comment says "caller can read normalized_clause_type" but the function provides no mechanism to retrieve the winning slug. Callers using this method have no actionable information beyond OTHER.

Use `classify_clause_type_string()` instead, but existing callers of `classify_clause_type()` are silently degraded.

#### F-09 — `probation` and `probation_period` are duplicate expected clauses with identical keywords
**File:** `backend/contract_profiles/employment.yaml`, `backend/contract_profiles/clause_keywords.yaml:168–172`
**Category:** Logic / Maintainability

The employment profile lists both `probation` (moved from optional in contract_fixes1) and `probation_period` in `expected_clauses`. Their keyword sets are identical: `probation`, `probationary period`, `trial period`. The `EXPECTED_CLAUSE_PATTERNS` dict in `contract_review_service.py` also has both:

```python
"probation": ["probation", "probation period", "probationary period", "trial period", "on probation"],
"probation_period": ["probation", "trial period", "probationary period"],
```

The same snippet can satisfy both, producing duplicate evidence blocks. `CANONICAL_CLAUSE_GROUPS` only normalizes the `compensation` family — `probation` and `probation_period` are not grouped, so the coverage matrix shows two separate entries for what is semantically one clause.

#### F-10 — Heuristic fallback loses all contract_type detection when Ollama is offline
**File:** `backend/services/document_classification_service.py:327–329`
**Category:** Logic / Edge Case

```python
# Heuristic fallback — no contract_type/jurisdiction detection
is_legal, confidence = self._keyword_classify(text)
return is_legal, confidence, None, True, None, None   # contract_type=None
```

When Ollama is offline, `_keyword_classify()` can determine `is_legal` correctly but always returns `contract_type=None`. The heuristic `_LEGAL_KEYWORDS` list contains employment-specific terms ("employee", "employer", "salary", "probation") that could trivially identify contract type, but this logic is absent. In offline/degraded environments, all documents are stored with `detected_contract_type=None`.

#### F-11 — DistilBERT is fed up to 3000 chars but truncates at 512 tokens
**File:** `backend/services/document_classification_service.py:422`
**Category:** Logic / Edge Case

```python
results = pipe(text[:3000], truncation=True, max_length=512)
```

3000 characters is approximately 600 GPT-style tokens. DistilBERT uses WordPiece tokenization and will truncate to 512 tokens, silently dropping roughly the last 15-20% of the sample. For Arabic/bilingual contracts where English contractual markers appear late in the document sample, these markers are dropped before classification. The sample should be sized to produce ≤512 tokens, or the truncation behavior should be explicitly documented.

#### F-12 — `msa.yaml` has `limitation_of_liability: high` risk weight but moved to optional
**File:** `backend/contract_profiles/msa.yaml`
**Category:** Logic / Consistency

`limitation_of_liability` was moved from `expected_clauses` to `optional_clauses` in contract_fixes1, but `risk_weights` still assigns it `high` severity. When this optional clause is not detected, the contract review engine generates a high-severity risk item for a clause that was explicitly categorized as optional. The risk weight should be lowered to `medium` or `low` for optional clauses.

#### F-14 — `_LEGAL_CATEGORY_PRIORITY` can diverge from `_CATEGORY_SLUG_KEYWORDS`
**File:** `backend/services/clause_taxonomy.py:15–76`
**Category:** Maintainability

`_LEGAL_CATEGORY_PRIORITY` (25 entries) and `_CATEGORY_SLUG_KEYWORDS` (25 entries) are independent structures. When a new clause category is added to `_CATEGORY_SLUG_KEYWORDS`, it is not automatically added to `_LEGAL_CATEGORY_PRIORITY`. Slugs absent from the priority list fall through to the `max()` fallback (line 272), bypassing the intended tie-break order. There is no validation asserting that both dicts are consistent.

#### F-15 — `" v "` and `"versus"` as case-file signals are too broad
**File:** `backend/services/document_classification_service.py:131`
**Category:** Edge Case / Assumptions

```python
" v ",   # padded to avoid matching inside words
"versus",
```

These signals count toward the `_CASE_FILE_THRESHOLD = 3` court-document gate. However, contracts contain these patterns legitimately: "Party A versus Party B", "the employee v. the employer's policy", liability comparisons ("option A v. option B"). If a contract already contains two other mild case-file signals (e.g., "judgment" in a clause referencing an arbitral judgment, "order" in an order form), hitting `" v "` pushes the count to 3 and the document is misclassified as `LEGAL_NON_CONTRACT`.

#### F-16 — `classify_termination_subtype()` is a first-match scan with order dependency
**File:** `backend/services/clause_taxonomy.py:330–333`
**Category:** Edge Case

```python
for subtype, keywords in subtype_keywords.items():
    if any(keyword in text_lower for keyword in keywords):
        return subtype  # first match wins
```

`TERMINATION_RIGHTS` is checked before `PROBATION_TERMINATION`. A clause about "right to terminate during probation" matches `TERMINATION_RIGHTS` first. This ordering is implicit and not documented. If the order of `subtype_keywords` changes (e.g., during refactoring), classification results change without any warning.

## Positive Notes

1. **Two-stage classification with graceful degradation** — `DocumentClassificationService` never raises; all paths return a valid `ClassificationResult`. The Ollama → heuristic fallback is well-structured.

2. **Case-file gate prevents DistilBERT misclassification** — The signal-count approach (≥3 signals → LEGAL_NON_CONTRACT) correctly handles court judgments before even running DistilBERT, avoiding false positives on the contract label.

3. **YAML-driven profiles** — `employment.yaml`, `nda.yaml`, `msa.yaml` with `contract_profile_loader.py` cache make adding new profiles trivial. The validation in `load_contract_profile()` catches malformed profiles early.

4. **Operative section inclusion of ANNEXURES_SCHEDULES** — The fix to include `ANNEXURES_SCHEDULES` in `operative_sections` correctly prevents SLA/IP clauses in Schedule-A or Annexure sections from being silently dropped.

5. **`_has_party_definitions()` covers all three profile types** — Employment (first/second party), MSA (service provider, vendor, client), and NDA (disclosing/receiving party) are all detected, preventing false-AMBIGUOUS classification on pages with party definitions.

6. **`CLAUSE_HEADING_SYNONYMS`** — The 17-entry synonym map correctly catches common heading variants (e.g., "Remuneration" → salary_wages, "Term of Agreement" → term) that primary keywords would miss.

7. **`IMPLICIT_GOV_LAW_KEYWORDS` / `IMPLICIT_DISPUTE_RESOLUTION_KEYWORDS`** — Post-pass implicit detection correctly handles GCC/Arabic contracts where governing law is referenced via statute citations rather than a standalone clause.

8. **SHA-256-based deterministic `clause_id`** — Clause IDs are content-addressable; the same clause text always generates the same ID, enabling deduplication and stable cross-run references.

## Recommendations (Priority Order)

1. **F-01 (Critical):** In `contract_review_service.py`, when `contract_type_key` is not in metadata, look up `document_registry.get_classification(contract_id)` and use `detected_contract_type` as the fallback before defaulting to `"employment"`. Surface this auto-detected type in the review response.

2. **F-02 (Critical):** Replace the fail-closed literal string check with a structural check (e.g., is the text structured as JSON or a numpy array?). Do not fail on the presence of common English words.

3. **F-03 (Critical):** Change the STATUTE detection condition from `"an act to" in sample or "section" in sample` to `"an act to" in sample` only (requiring the specific statutory preamble phrase).

4. **F-05 (High):** Create a `backend/contract_profiles/other.yaml` with a minimal set of universal expected clauses (governing_law, termination, liability), or map `"other"` to `"employment"` as an explicit fallback in `contract_review_service.py`.

5. **F-06 (High):** Replace the hardcoded warning string with an f-string that includes `contract_type_key`.

6. **F-07 (High):** Remove `r"@\w+\.\w+"` from `administrative_markers` or limit it to pages where email is the primary content (e.g., the regex only triggers if the line contains only an email-like string).

7. **F-09 (Medium):** Remove `probation` from `expected_clauses` in `employment.yaml` (keeping only `probation_period`), or add `["probation", "probation_period"]` to `CANONICAL_CLAUSE_GROUPS`.

8. **F-12 (Medium):** Lower `limitation_of_liability` risk weight in `msa.yaml` from `high` to `low` or `medium` to reflect its optional status.
