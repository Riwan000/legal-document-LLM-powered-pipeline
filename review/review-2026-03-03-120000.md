## Summary

The changeset is a well-structured set of 9 improvements to the Contract Review pipeline (risk scoring, contradiction detection, LLM uncertain-clause resolver, statutory references, PDF rewrite, and a new admin clear-all endpoint). The overall architecture is sound, error handling is defensive, and the UI additions are coherent. However, there are several bugs and design problems—one of which is a critical security issue—that should be addressed before merging.

## Findings

### CRITICAL

#### C1 — Unauthenticated destructive endpoint: `POST /api/admin/clear-all`
**File:** `backend/main.py:1512–1591`

The new endpoint permanently deletes all indexed documents, vector embeddings, clauses, and on-disk files. It requires no authentication token, API key, role check, or IP restriction. Any HTTP client with network access to port 8000 can invoke it—including via a malicious web page exploiting the `allow_origins=["*"]` CORS policy.

```python
@app.post("/api/admin/clear-all")
async def clear_all_documents():
    ...
    for doc_id in doc_ids:
        vector_store.delete_document(doc_id)
        clause_store.delete_document(doc_id)
        ...
```

The UI confirmation checkbox (`confirm_clear_all`) runs client-side only and is irrelevant to direct API calls.

**Recommended fix:** Add a `Bearer` token check, a secret env-var header, or restrict to `127.0.0.1` in an `Depends()`. At minimum, add a `POST`-only body parameter that must equal an env-var secret:
```python
secret: str = Form(...)
if secret != settings.ADMIN_SECRET:
    raise HTTPException(status_code=403, detail="Forbidden")
```

### HIGH

#### H1 — `_is_likely_operative_contract` rejects NDAs and MSAs silently
**File:** `backend/services/contract_review_service.py:436–449`

The document classification function returns `False` (triggering a "document may not be an operative contract" warning) when neither `"employer"` nor `"employee"` appears in the first 5,000 characters:

```python
if "agreement" not in lead or ("employer" not in lead and "employee" not in lead):
    return False
```

An NDA or MSA will almost never contain "employer"/"employee" in the preamble. This means Contract Review on NDA/MSA documents will systematically show a misleading warning that may cause reviewers to distrust the output.

**Fix:** The party-term check should be profile-aware, or use a wider set of party terms:
```python
PARTY_TERMS = {"employer", "employee", "disclosing party", "receiving party",
               "service provider", "client", "vendor", "contractor"}
if "agreement" not in lead or not any(t in lead for t in PARTY_TERMS):
    return False
```

#### H2 — Inconsistent `FileParser.parse_file` tuple unpacking across the same method
**File:** `backend/services/contract_review_service.py:1580–1581` vs `962–1002`

In the document classification pre-check (line 1580–1581), pages are unpacked as 2-tuples:
```python
pages = FileParser.parse_file(Path(doc_path))
full_text = "\n".join((t or "") for t, _ in pages)
```

But in `_page_fallback_scan` (line 998–1002), the same `FileParser.parse_file` output is guarded against both 2 and 3-tuple lengths:
```python
if len(page_entry) == 3:
    page_text, page_num, _ = page_entry
else:
    page_text, page_num = page_entry
```

If `FileParser.parse_file` returns 3-tuples (page_text, page_num, metadata), the 2-tuple unpack on line 1581 raises `ValueError: too many values to unpack` and silently swallows the exception on line 1587, skipping the classification check. The two call sites need to use the same unpacking strategy.

### MEDIUM

#### M1 — `get_stats()` triggering disk I/O on every sidebar refresh
**File:** `backend/services/vector_store.py:680`

`get_stats()` now calls `self._ensure_loaded()`, meaning the first call from the admin panel loads the entire FAISS index from disk. For large deployments the index can be hundreds of MB. The admin sidebar will now appear to "hang" on first access since `_ensure_loaded()` is synchronous and blocks the event loop if called from an async route without `run_in_executor`.

The original decision to *not* call `_ensure_loaded()` in `get_stats()` was explicit and documented (removed comment). If accurate counts are needed for the admin panel, consider a separate lightweight metadata count stored alongside the index file.

#### M2 — G8 LLM resolver silently no-ops when `clause_ids` is empty
**File:** `backend/services/contract_review_service.py:1882–1893`

```python
_g8_ev_text = next(
    (c.get("text", "") for c in evidence_candidates
     if c.get("clause_id") in (presence_map[_g8_clause_type].get("clause_ids") or [])),
    "",
)
```

For clauses that hit `uncertain` via the keyword match (not heading match), `presence_map[clause_type]["clause_ids"]` is set to `[result["clause_id"]]`. However, if `result["clause_id"]` is `None` (which happens when the match came from a candidate without a `clause_id` field), `clause_ids` becomes `[None]` and `c.get("clause_id") in [None]` evaluates to `True` only if `c.get("clause_id")` is also `None`.

The consequence is that the LLM resolver gets an empty `_g8_ev_text` and asks Ollama "Does the following text contain a X clause? Reply only: YES or NO.\n\n" — always getting `not_detected`, potentially downgrading a legitimately uncertain clause.

**Fix:** Filter out `None` from `clause_ids`:
```python
valid_ids = [cid for cid in (presence_map[_g8_clause_type].get("clause_ids") or []) if cid]
_g8_ev_text = next(
    (c.get("text", "") for c in evidence_candidates if c.get("clause_id") in valid_ids),
    "",
)
```

#### M3 — Duplicate display names: `probation` and `probation_period` both map to "Probation Period"
**File:** `backend/services/contract_review_service.py:214, 217`

```python
"probation":               "Probation Period",       # line 214
"probation_period":        "Probation Period",        # line 217
```

Both `_LEGAL_CATEGORY_PRIORITY` and `EXPECTED_CLAUSE_PATTERNS` contain both keys (`probation` and `probation_period`) with overlapping keyword sets. If both appear in the same contract profile's `expected_clauses`, the coverage matrix will show two rows with the same display name "Probation Period", which is confusing. The `CANONICAL_CLAUSE_GROUPS` map should group them similarly to `compensation`/`salary_wages`.

#### M4 — `CANONICAL_CLAUSE_GROUPS` references `"remuneration"` which is not a valid clause key
**File:** `backend/services/contract_review_service.py:240–243`

```python
CANONICAL_CLAUSE_GROUPS: Dict[str, List[str]] = {
    "compensation": ["compensation", "salary_wages", "remuneration"],
}
```

`"remuneration"` is not a key in `EXPECTED_CLAUSE_PATTERNS`, `EXPECTED_CLAUSE_DISPLAY_NAMES`, or any shipped profile's `expected_clauses`. In `_aggregate_status`, it is filtered out silently by:
```python
member_keys = [m for m in members if m in expected]
```
This is safe but misleading — `remuneration` appears in the group definition but never applies.

#### M5 — `_classify_evidence_label` is nearly useless for non-employment documents
**File:** `backend/services/contract_review_service.py:843–857`

The semantic labeler handles only two cases (death compensation and termination/notice). For all NDA, MSA, and IP-related clauses it returns `None`, causing `display_name` to fall through to the generic `"Clause (Page N)"` format. Given that `_evidence_block_display_name` already falls back gracefully, the function adds negligible value in its current state and should either be extended or removed to reduce dead code.

#### M6 — Admin clear-all: file deletion happens *after* registry deletion
**File:** `backend/main.py:1555–1570`

```python
try:
    document_registry.delete_document(doc_id)
except Exception as e:
    ...

for rec in (versions or []):
    file_path = rec.get("file_path")
    ...
    p.unlink()
```

`versions` is fetched before the registry is deleted (line 1529), so the data is available regardless. But if `registry.delete_document` raises, execution continues to the file deletion loop. Files will be physically deleted but registry records may still exist, leaving dangling pointers. The order should be: delete files → delete registry.

### LOW

#### L1 — `_CORSMiddlewareMarker` multiple-inheritance hack is fragile
**File:** `backend/main.py:97–108`

```python
class _CORSMiddlewareMarker(StarletteMiddleware, CORSMiddleware):
    pass
```

This class is created solely to satisfy a test's `type(m) == CORSMiddleware` assertion. The comment acknowledges this. The right fix is to update the test, not to introduce a production-class hierarchy that carries undefined behavior from dual-inheriting Starlette `Middleware` (a dataclass/descriptor) and `CORSMiddleware` (an ASGI callable).

#### L2 — `DISPLAY_NAME_MAP` uses fragile concatenated keys
**File:** `backend/services/contract_review_service.py:331–348`

Keys like `"confidentialinformat"` and `"governinglaw"` are derived by stripping non-alphanumeric chars from `clause_id` segments. If the clause ID generation scheme changes (e.g., prefix format), these lookups silently fail and fall back to the generic title-case display name with no warning.

#### L3 — `classify_clause_type_string` silently prefers enum score over string score
**File:** `backend/services/clause_taxonomy.py:229`

```python
if slug not in all_scores:   # skip if already scored by enum path
    all_scores[slug] = score
```

If both enum-backed and string-keyed dictionaries have entries for `"governing_law"` with different keyword sets, the string-keyed keywords are completely ignored. The comment says "skip if already scored", but higher string-keyed scores would be discarded. This could lead to category misclassification if the YAML file overrides with richer keywords.

#### L4 — PDF contradiction risk section has orphaned partial code
**File:** `frontend/app.py` (new UI block for contradiction risks)

In the diff, the Streamlit section for contradiction risks begins with:
```python
for cr in contradiction_risks:
    if isinstance(cr, dict):
        st.info(f"ℹ️ {cr.get('description', '')}")

        # Verbatim evidence with keyword highlighting (Group 4b)
                kw = snippet.get("matched_keyword") or ""
```

The indented `kw = snippet.get(...)` block appears to be a fragment of the evidence expander that was not fully integrated—it is indented as if inside a loop that was removed, and `snippet` is undefined at that scope. This will cause a runtime `NameError` when a contradiction risk with evidence snippets is rendered.

#### L5 — `_extract_duration` in contradiction detection extracts only the first match
**File:** `backend/services/contract_review_service.py:1140–1150`

`re.search` finds only the first duration value. A clause saying "30 days probation; after that, 90 days notice" yields `30` days, not `90`. For notice period consistency checks, the *last* or *maximum* duration is often the operative one. This is a heuristic limitation, but should be documented.

## Positive Notes

- **Defensive LLM wrapper:** `_resolve_uncertain_with_llm` catches all exceptions, logs at DEBUG, and returns `"uncertain"` — never raises to the caller. This is the correct pattern for an optional AI enhancement.
- **`enforce_non_prescriptive_language` guardrail** is applied consistently to every generated text field (risk descriptions, executive summary), ensuring no prescriptive language leaks into output.
- **`WEAK_EVIDENCE_RULES` sentinel pattern:** using `[]` to mean "always `detected`" (no weak-evidence check) is elegant and self-documenting.
- **ANNEXURES_SCHEDULES inclusion:** the fix to include annexures/schedules in `operative_sections` correctly unblocks SLA/IP clause detection in Schedule-A attachments.
- **`_page_fallback_scan` is non-fatal:** all errors are caught and logged; the function returns `{}` silently, so a corrupt or missing file does not crash the review pipeline.
- **Confirmation checkbox** in the admin panel provides a useful human safeguard for the clear-all action (even though it's bypassable via direct API call).
- **`model_dump()` used consistently** when storing evidence in `ctx.add_result`, ensuring Pydantic serialization correctness.
- **`_build_key_review_observations` sort order** (`risk → finding → confirmation`, then severity) is deterministic and reviewer-friendly.
