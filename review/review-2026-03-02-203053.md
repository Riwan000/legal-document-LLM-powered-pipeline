#### CR-1 · `extract_clauses()` does not exist — `AttributeError` at runtime
**File:** `backend/services/contract_review_service.py:1643`
**Category:** Logic Bug

```python
raw_clauses = self.clause_extractor.extract_clauses(file_path, contract_id, use_structured=True)
```

The `StructuredClauseExtractor` class exposes `extract_structured_clauses()` (defined at `backend/services/structured_clause_extraction.py:265`). There is no `extract_clauses()` method. Every call to the contract review pipeline crashes here with `AttributeError: 'StructuredClauseExtractor' object has no attribute 'extract_clauses'`.

**Fix:** Rename the call to `self.clause_extractor.extract_structured_clauses(...)` and check whether `use_structured` is an accepted parameter.

**File:** `backend/services/contract_review_service.py:1893–1920`
**Category:** Logic Error

`_skip_risk_statuses` at line 1893 explicitly includes `"uncertain"`. The `continue` at line 1900 means control never reaches line 1911. Lines 1911–1915 generate the "weak evidence" risk description for uncertain clauses, but this branch is permanently unreachable:

```python
_skip_risk_statuses = (
    "detected", "detected_implicit", "detected_distributed", "detected_weak",
    "implicitly_covered",
    "uncertain",   # <-- skipped here ...
)
for clause_type in expected:
    status = presence_map.get(clause_type, {}).get("status", "not_detected")
    if status in _skip_risk_statuses:
        continue          # uncertain hits this continue
    ...
    if status == "uncertain":   # <-- never reached
        description = f"Clause detected with weak evidence signal: ..."
```

Uncertain clauses are surfaced via `implicitly_covered_clauses` at line 2072, which is correct. The orphaned risk-row branch at 1911–1920 should be removed.

### HIGH — Wrong Behavior / Silent Data Loss

---

#### H-1 · Silent YAML load failure in `clause_taxonomy.py`
**File:** `backend/services/clause_taxonomy.py:82–94`
**Category:** Silent Failure

```python
try:
    with open(path, encoding="utf-8") as f:
        return yaml.safe_load(f) or {}
except Exception:
    pass
return {}
```

Any YAML parse error or missing file returns an empty dict with no log message. `_CLAUSE_KEYWORDS` becomes `{}`, `string_type_keywords` is never populated, and all profile-driven clause type detection silently stops working. A single corrupt YAML disables a significant portion of taxonomy.

**Fix:** At minimum, log the exception at `WARNING` or `ERROR` level before falling through to `return {}`.

---

#### H-2 · Reverse-substring false positive in heading-match scan
**File:** `backend/services/contract_review_service.py:1866`
**Category:** Logic Error

```python
if norm_heading_kw in title_norm or title_norm in norm_heading_kw:
```

The second condition (`title_norm in norm_heading_kw`) is inverted. If `title_norm = "a"` (a common letter or short section title) and `norm_heading_kw = "salary"`, the condition fires because `"a" in "salary"` is `True`. Short or generic clause headings trigger false `detected_implicit` promotions across unrelated clause types.

**Fix:** Remove the second condition. Only the first direction (`norm_heading_kw in title_norm`) makes semantic sense — check that the keyword appears in the clause title, not the reverse.

---

#### H-3 · `_extract_duration()` captures only the first number — false contradiction alerts
**File:** `backend/services/contract_review_service.py:1140–1148`
**Category:** Assumption Violation

`re.search` returns the first match in the text. For clauses like _"notice of at least 30 days, but not more than 90 days"_, `term_days=30` and `notice_days=90` produces a ratio of 3.0 and raises a contradiction risk even though the two clauses are compatible. The same number appearing in unrelated sentences (e.g. "the employer shall pay within 30 days") can pair with a different clause's duration to produce spurious contradictions.

**Fix:** Use `re.findall` and select the value closest to a `termination|notice` keyword, or restrict the search to a shorter evidence window.

---

#### H-4 · `_attach_verbatim_evidence()` skips uncertain clauses that have evidence
**File:** `backend/services/contract_review_service.py:1260`
**Category:** Logic Error

```python
if risk.status in ("not_detected", "uncertain") or risk.missing_clause:
    continue
```

The docstring on line 1252 says it skips uncertain and not-detected, but per the pipeline design, uncertain clauses *do* have candidate evidence (they reached `uncertain` because weak evidence was found). Skipping them means the UI can never show verbatim evidence for an uncertain clause, depriving reviewers of the very text that explains why detection was ambiguous.

#### H-5 · `classify_clause_type_string()` raises `ValueError` on empty input
**File:** `backend/services/clause_taxonomy.py:226–228`
**Category:** Edge Case

```python
if not all_scores:
    return "other"
return max(all_scores.items(), key=lambda x: x[1])[0]
```

This is actually handled correctly when `all_scores` is empty (returns `"other"`). However, `all_scores[clause_type.value] = score` at line 218 overwrites previous enum-keyed scores rather than accumulating them if the same `.value` string appears twice. If `ClauseType` has duplicate `.value` strings (e.g., two members map to `"other"`), the last score wins silently. Not a crash, but a silent scoring bug worth knowing.

### MEDIUM — Logic Errors & Edge Cases

---

#### M-1 · Contradiction check fires when clauses define different *types* of duration
**File:** `backend/services/contract_review_service.py:1165–1167`
**Category:** Assumption Does Not Hold

The ratio threshold (`> 1.5`) assumes that both extracted durations refer to the same thing (e.g., both are notice periods). But `term_text` may contain a contract duration (e.g., "1 year = 365 days") while `notice_text` contains a notice period ("30 days"). Ratio = 12.2 → false contradiction every time.

---

#### M-2 · Page-number `None` not guarded before comparison in clause extraction
**File:** `backend/services/structured_clause_extraction.py:308–311`
**Category:** Edge Case

```python
if page_number <= 0:
    continue
```

If `page_number` is `None` (possible if the parser returns a page tuple with a missing number), this raises `TypeError: '<=' not supported between instances of 'NoneType' and 'int'`.

**Fix:** `if not page_number or page_number <= 0: continue`

---

#### M-3 · `presence_map` initialised with scalar returns but treated as lists
**File:** `backend/services/contract_review_service.py:1747–1752`
**Category:** Type Mismatch

`_detect_clause_presence()` returns a single `clause_id` and `page_number`. The presence map wraps these in lists at construction time, which is correct. But callers at line 1903 use `(entry.get("clause_ids") or [None])[0]` — if the wrapper was applied only once at init but `_detect_clause_presence` was called multiple times and results were appended differently elsewhere, the list vs scalar assumption breaks. Needs an explicit invariant check or typed dataclass.

---

#### M-4 · `clause_keywords.yaml` conflates non-solicitation and non-compete keywords
**File:** `backend/contract_profiles/clause_keywords.yaml:174–178`
**Category:** Data Quality

`non_solicitation` contains `"non compete"`. Non-compete and non-solicitation are legally distinct clauses; conflating their keywords causes incorrect clause-type detection and mislabels risk items in the review report.

---

#### M-5 · `jurisdiction_statutes.yaml` — incomplete coverage silently returns no notes
**File:** `backend/legal_references/jurisdiction_statutes.yaml`
**Category:** Assumption Does Not Hold

The YAML covers employment-only clauses (notice, leave, probation, gratuity) for 6 GCC jurisdictions. NDA and MSA contracts that invoke GCC jurisdiction will receive no `statutory_notes` in the response — silently, with no indication that the YAML simply doesn't cover those clause types. Users may believe no statutes exist.

---

#### M-6 · Jurisdiction `_canon_jurisdiction()` misses `"Saudi Arabia"` variant
**File:** `backend/services/contract_review_service.py:1438`
**Category:** Edge Case

The normalisation map covers "kingdom of saudi arabia" and "ksa" but not "Saudi Arabia" or "saudi arabia" as standalone strings (no "kingdom" prefix). Contracts that write "Governing Law: Saudi Arabia" get no jurisdiction statutory notes.

---

#### M-7 · `title_norm in norm_heading_kw` also affects short numeric titles
**File:** `backend/services/contract_review_service.py:1866`
**Category:** Regression Risk (related to H-2)

If a clause heading is `"1."` or `"a"` (common in Arabic/bilingual documents), nearly all `norm_heading_kw` strings contain that substring. This causes mass false promotions of undetected clauses to `detected_implicit` status, inflating the detected clause count.

---

#### M-8 · `_load_clause_keywords()` silent failure also affects `category_keywords` init
**File:** `backend/services/clause_taxonomy.py:109–162`
**Category:** Related to H-1

`string_type_keywords` is populated from `_CLAUSE_KEYWORDS` (line ~120). If `_CLAUSE_KEYWORDS` is empty (silent load failure), `string_type_keywords` is `{}`. Subsequently `classify_clause_type_string()` only uses enum-backed types, silently dropping all profile-specific clause types without any error or warning. This is a runtime regression that's very hard to detect in logs.

#### L-1 · Hard-coded keyword lists and YAML keywords are duplicative
**File:** `backend/services/clause_taxonomy.py:115–162` and `backend/contract_profiles/clause_keywords.yaml`
**Category:** Maintainability

The `_CATEGORY_SLUG_KEYWORDS` dict in `clause_taxonomy.py` contains the same keyword patterns as `clause_keywords.yaml`. If they diverge, the taxonomy service uses YAML for `string_type_keywords` and the hard-coded dict for `category_keywords`, producing split and potentially conflicting classification results.

**Category:** Dead Code

Aliases (e.g. `"compensation"` → `"compensation_benefits"`) are added to `extracted_types` (line 1659), but no downstream matching logic reads from `extracted_types` for alias-expanded lookups. The set is only used for logging/coverage at lines 2005–2015.

**Category:** Dead Field

`self.is_definition: bool = False` is never set to `True` anywhere. No code checks it. It is either an unimplemented feature or leftover stub that should be removed or implemented.

#### L-4 · `unit_type` field on `ExtractedClause` is always `"clause"`
**File:** `backend/services/structured_clause_extraction.py:78`

`self.unit_type: str = "clause"` is never changed, even though the comment on line 74 implies it should distinguish `"clause"`, `"definition"`, and `"page_chunk"`. **File:** `frontend/app.py:258`
**Category:** Bad Practice

```python

```

Bare `except` swallows `KeyboardInterrupt`, `SystemExit`, and `GeneratorExit`. Should be `except Exception:`.

#### L-6 · `jurisdiction_statutes.yaml` inconsistent units across jurisdictions
**File:** `backend/legal_references/jurisdiction_statutes.yaml`
**Category:** Data Quality

KSA and UAE express gratuity in *days* per year; Bahrain expresses it in *months* per year. Consuming code that formats these notes for display may produce confusing or unit-inconsistent output without an explicit unit field per entry.

## Positive Notes

- **Fail-safe architecture** — `_resolve_uncertain_with_llm()` (line 1097) wraps all Ollama errors and returns `"uncertain"` on failure, preventing LLM unavailability from crashing the review.
- **YAML-driven taxonomy** — Moving clause keywords to `clause_keywords.yaml` is the right direction; profile-driven loading (Group 1a) is a significant improvement over hard-coded enums.
- **`enforce_non_prescriptive_language()`** is used consistently throughout risk description generation, enforcing uniform LLM output formatting.
- **`_identify_contradiction_risks()`** is a genuinely useful feature. The current two-clause check (termination vs. notice) is a good MVP for the contradiction detection use case.
- **Evidence guardrails** and two-pass validation in `EvidenceGuardrailService` are well-structured and add meaningful safety to LLM responses.
- **PDF generation** (Group 8) is thorough, with TOC, coverage table, evidence cells, and statutory reference section all properly structured.
- **G8 LLM uncertain-resolver** is correctly gated to `standard` depth and properly falls back to `"uncertain"` on failure — safe and non-blocking.
